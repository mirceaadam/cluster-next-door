What if your neighbor's server room
was your DR site?

Two houses. Two racks. One cluster.

The core idea: what if high availability didn't need multiple cloud regions? What if it just needed a good neighbor and some fiber?

High Availability is Expensive 💸

  • Multi-AZ EKS: ~$150-300/month minimum
  • Multi-region: 2-3x that cost
  • DR patterns assume cloud-native = cloud-hosted
  • Small teams can't justify the spend

Enterprise HA patterns weren't designed for homelabbers.

Cloud HA is designed for companies with budgets. Even a basic multi-AZ setup costs real money. What about the rest of us?

What if we did it differently? ⚡

  • ☀️ Solar-powered nodes (daytime compute is free)
  • ♻️ Recycled enterprise hardware (ThinkPads, decommissioned servers)
  • 🔌 Tailscale mesh between houses (no port forwarding)
  • ☁️ AWS as cold DR only (EC2 stopped = $0)

Total monthly cost: ~$0.50

Solar panels, recycled hardware, and a mesh VPN. The only recurring cost is S3 for WAL backups.

Karmada Federation 🌍

Distribute workloads across clusters with policy-based scheduling

apiVersion: policy.karmada.io/v1alpha1
kind: PropagationPolicy
metadata:
  name: figurines-propagation
spec:
  conflictResolution: Overwrite
  resourceSelectors:
    - apiVersion: apps/v1
      kind: Deployment
      name: figurines-frontend
  placement:
    clusterAffinity:
      clusterNames:
        - k3s-m
        - k3s-c
    replicaScheduling:
      replicaSchedulingType: Duplicated
  failover:
    application:
      decisionConditions:
        tolerationSeconds: 120
      purgeMode: Never

Duplicated = full copy on each site. Cloudflare handles routing.

Karmada's PropagationPolicy is the key. Duplicated scheduling means each cluster gets a full deployment. Failover is handled at the DNS/tunnel level by Cloudflare.

Cloudflare Tunnels 🚇

  • Zero port forwarding required
  • Cloudflared pod per cluster (outbound only)
  • Automatic failover between connectors
  • Free tier — no cost

How failover works:
Cloudflare sees multiple connectors for the same tunnel.
If one connector goes down → traffic routes to the healthy one automatically.

⚠️ Gotcha: Use separate tunnels per cluster for true DR (more on this later)

Cloudflare Tunnels are magic for homelabs. No firewall rules, no dynamic DNS, no NAT traversal headaches. The 404 gotcha is real — shared tunnels can conflict.

CloudNativePG 🐘

Primary (k3s-m)

  • Write traffic
  • Streaming replication out
  • WAL archiving to S3

Replica (k3s-c)

  • Read-only traffic
  • Hot standby
  • Promote on primary failure

Cross-site replication over Tailscale WireGuard tunnel
S3 WAL backups = point-in-time recovery for ~$0.50/month

CloudNativePG handles Postgres natively in Kubernetes. Streaming replication works beautifully over Tailscale. The S3 WAL backup is our last-resort DR.

External Secrets Operator 🔐

Zero-cost secret management with AWS Parameter Store

apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: app-secrets
  namespace: figurines
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: aws-parameter-store
    kind: ClusterSecretStore
  target:
    name: app-secrets
  data:
    - secretKey: DATABASE_URL
      remoteRef:
        key: /cluster-next-door/figurines/database-url
    - secretKey: CLOUDFLARE_TOKEN
      remoteRef:
        key: /cluster-next-door/cloudflare/tunnel-token

No more kubectl create secret on every cluster 🎉

ESO + Parameter Store is completely free and eliminates manual secret management. Secrets sync automatically across all clusters.

Failover Demo 🔥

What happens when House A goes down?

  1. Cloudflare detects connector failure (~30s)
  2. Traffic routes to House B connector
  3. Users see zero downtime
  4. Karmada marks cluster as NotReady (~120s)
  5. When House A returns → traffic rebalances

🎯 RTO: ~30 seconds | RPO: 0 (active-active)

The real magic — let's pull the power on one site and watch Cloudflare failover in real-time. RTO is about 30 seconds, RPO is zero because both sites are active.

What Went Wrong 💥

  • Karmada operator: add — doesn't append, it replaces. Lost env vars on multiple deploys before figuring this out.
  • Cloudflared shared tunnel 404s — two clusters sharing one tunnel token causes route conflicts. Random 404s that are impossible to debug.
  • LXC → VM migration — started with LXC containers, but k3s needs kernel features that LXC doesn't provide cleanly. Had to rebuild everything as VMs.
  • Manual secrets — kubectl create secret on 4 clusters is not fun. ESO fixed this.
Lessons from production. The Karmada add operator is particularly sneaky — the name implies append but it actually replaces. Cost me hours of debugging.

What I'd Do Differently 🔄

  • ✅ Start with ESO from day 1 (secrets across clusters are painful)
  • ✅ Separate Cloudflare tunnels per cluster from the start
  • ✅ VMs over LXCs for k3s (save the debugging time)
  • ✅ Set conflictResolution: Overwrite in every PropagationPolicy
  • ✅ Public Grafana dashboards need panel IDs — plan JSON structure early
If I started over today, these are the things I'd do from the beginning. Each one cost me at least half a day of debugging.
