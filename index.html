Karmada — Multi-Cluster Federation ☸️

  • What: CNCF project that turns multiple K8s clusters into one logical platform
  • How: You define workloads once → Karmada distributes them based on policies
  • Why: No manual kubectl apply on each cluster — one API, many clusters

In this project: Karmada runs on k3s-karmada (L570) and pushes the figurines + dashboard app to both k3s-m and k3s-c using Duplicated scheduling — full copy on each site.

Karmada is like a scheduler on top of schedulers. You write normal Kubernetes manifests, add a PropagationPolicy, and Karmada handles placing them across clusters. In our case both sites get a full copy so either can serve traffic independently.

Karmada Federation 🌍

Distribute workloads across clusters with policy-based scheduling

apiVersion: policy.karmada.io/v1alpha1
kind: PropagationPolicy
metadata:
  name: dashboard-propagation
spec:
  resourceSelectors:
    - apiVersion: apps/v1
      kind: Deployment
  placement:
    clusterAffinity:
      clusterNames: [k3s-m, k3s-c]
    replicaScheduling:
      replicaSchedulingType: Duplicated
  failover:
    application:
      decisionConditions:
        tolerationSeconds: 120
      purgeMode: Never

Duplicated = full copy on each site. Cloudflare handles routing.

Karmada is the CNCF federation project. PropagationPolicy tells it where to place workloads. Duplicated means both clusters get a full copy. purgeMode Never means don't evict — let Cloudflare handle the routing decision.

Resilience in Action 🛡️

What happens during a network outage or hardware failure?

  1. Cloudflare detects connector failure (~30s)
  2. Traffic routes to the healthy site automatically
  3. Users experience zero downtime
  4. Karmada marks affected cluster as NotReady (~120s)
  5. Once connectivity restores → traffic rebalances

🎯 RTO: ~30s | RPO: 0 (active-active)

This is the core value prop. Active-active across two houses. Cloudflare does the fast failover (30s). Karmada is aware but doesn't reschedule — we use purgeMode: Never because both sites already have the app.

Cloudflare Tunnels 🚇

  • Zero port forwarding required
  • Cloudflared pod per cluster (outbound only)
  • Automatic failover between connectors
  • Free tier — no cost

Failover: Cloudflare sees multiple connectors for the same tunnel. If one goes down → traffic routes to the healthy one.

Key insight: no ports open on either router. Cloudflared makes outbound connections only. ISP blocks port 443 anyway (ZTE F668). Free tier covers everything we need. Separate tunnel per cluster avoids the 404 issue we had before.

CloudNativePG is a CNCF operator for PostgreSQL. Primary on k3s-m handles writes. Replica on k3s-c at Catalin's gets streaming replication over Tailscale. WAL segments archived to S3 for point-in-time recovery. Total storage cost: about 50 cents a month.

External Secrets Operator 🔐

Zero-cost secret management with AWS Parameter Store

apiVersion: external-secrets.io/v1
kind: ExternalSecret
metadata:
  name: app-secrets
spec:
  refreshInterval: 24h
  secretStoreRef:
    name: aws-parameter-store
    kind: ClusterSecretStore
  data:
    - secretKey: DATABASE_URL
      remoteRef:
        key: /nk/my-app/DATABASE_URL

No more kubectl create secret on every cluster 🎉

ESO syncs secrets from AWS Parameter Store (free tier) into Kubernetes secrets on every cluster. One source of truth. No more manually creating secrets on 4 clusters. Refreshes every 24h automatically.

What Went Wrong 💥

  • Karmada operator: add — replaces, doesn't append. Lost env vars.
  • Shared tunnel 404s — 3 connectors, only 1 has all Ingresses. Random 404s.
  • LXC → VM — k3s needs kernel features LXC can't provide. Rebuilt everything.
  • Manual secrets — kubectl create secret on 4 clusters. ESO fixed this.
  • Unstable Deployments Deployments were a hit / or miss
War stories. Karmada add operator silently replaces arrays. Shared Cloudflare tunnel caused random 404s because only one connector had the ingress routes. LXC to VM migration was painful — had to rebuild everything. Manual secrets on 4 clusters was unsustainable.

What I'd Do Differently 🔄

  • ✅ Start with ESO from day 1
  • ✅ Successful "Template Application" used as example
  • ✅ Separate Cloudflare tunnels per cluster
  • ✅ VMs over LXCs for k3s
  • ✅ conflictResolution: Overwrite in every PropagationPolicy
Hindsight is 20/20. ESO would have saved weeks of manual secret management. Template app pattern means new services deploy in minutes. Separate tunnels fixed the 404 lottery. VMs from day 1 would have avoided the LXC rebuild. And always set conflictResolution Overwrite or Karmada silently ignores existing resources.
